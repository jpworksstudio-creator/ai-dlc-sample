# プロダクト仮説

## 対象ユーザー

4人家族（父・母・2歳の子）のうち、**父**が主利用者。家計の把握・改善を担い、Zaim または Moneytree で記録した家計データをエクスポートして利用する。

## 現在の行動・業務

- Zaim、Moneytree 等の家計簿アプリで家計を記録している
- 先月の収支**総額**だけを確認している
- 支出の内訳や明細の正確性は、1件ずつ手動で確認している
- クレジットカード明細を見ても、何を購入したかがすぐ判別できない

## 困っていること

- 先月「何にいくら使ったか」がすぐ確認できない
- クレジットカード明細から購入内容をすぐ判別できない
- 購入情報のカテゴリ分けが間違っていることがある（例：おむつ代が食費に分類され、実際の食費が把握できない）

## 現在の代替手段

- 家計簿アプリで収支総額のみ確認
- 明細の金額・カテゴリが正しいか、手動で1件ずつ確認

## 理想的な体験

AI エージェントに自然言語で質問すると、補正済みの先月の支払い明細がわかる。無駄な支払いの指摘や家計の改善案を提案してもらえる。良かった点の評価も受けられる。

## 最も危険な仮説

H-1: 家計簿アプリのエクスポートデータに含まれる**金額・カテゴリの誤り**を補正し、ユーザーが意図したカテゴリ（例：食費）で正確な支出額を把握できる。

H-2: クレジットカード明細のような判別しづらい取引内容を、ユーザーが理解できる形に整理できる。

## 成功指標

SM-1: 代表質問5種類のうち4種類以上で30秒以内に応答が返る

SM-2: 食費・日用品・固定費の3カテゴリについて、手動補正後の金額と±10%以内

SM-3: 家計相談10回中8回以上で、改善案1件以上＋良かった点1件以上が含まれる

## 対象外の体験

- ネイティブモバイルアプリ
- 複数世帯・複数ユーザー管理（ユーザー申告）
- 投資・資産運用アドバイス（ユーザー申告）
- レシート OCR / 自動仕訳（ユーザー申告）
- その他の対象外（ユーザー申告：E を選択、詳細は未記入）
- 家計簿アプリとの API 自動連携（エクスポート取り込み前提のため対象外）

## 事実

- プロダクトディスカバリーを 2026-06-07 に開始した
- 小規模グリーンフィールド MVP として分類する（ユーザー申告）
- 想定世帯は4人家族（父・母・2歳の子）、主利用者は**父**（ユーザー申告）
- MVP で対応する家計簿アプリは **Zaim** と **Moneytree**（ユーザー申告）
- プラットフォームは Web アプリ、利用環境はスマホブラウザ（ネイティブアプリではない）（ユーザー申告）
- エクスポートした家計データは**端末内（ブラウザローカル）のみ**保存する（ユーザー申告）
- 「良かった点の評価」は MVP **必須**機能（改善案とセット）（ユーザー申告）
- 成功指標 SM-1〜3 に定量的閾値を設定した（ユーザー申告）
- 構築（Construction）・コード生成・インフラ構築は本フェーズの対象外（ユーザー申告）

## 仮説

- H-3: Zaim / Moneytree のエクスポート（CSV 等）だけで、MVP に必要な取引データを十分に取得できる
- H-4: カテゴリ誤りの主因は自動分類の精度であり、明細内容の判別困難さと組み合わさって把握を妨げている
- H-5: 自然言語での質問応答が、手動で1件ずつ確認するより負担が低く、家計改善の行動につながる
- H-6: 改善案と良かった点の評価をセットで提供すると、父（主利用者）の家計改善モチベーションが維持しやすい

## 決定事項

- D-1: プラットフォームは Web アプリとする（スマホブラウザ利用）
- D-2: ネイティブアプリは作らない
- D-3: データ取得は家計簿アプリからのエクスポートを前提とする（API 直接連携は MVP では行わない）
- D-4: 主利用者は父とする
- D-5: MVP で対応する家計簿アプリは Zaim と Moneytree とする
- D-6: 「良かった点の評価」は MVP 必須とする（改善案とセット）
- D-7: 家計データは端末内（ブラウザローカル）のみに保存する
