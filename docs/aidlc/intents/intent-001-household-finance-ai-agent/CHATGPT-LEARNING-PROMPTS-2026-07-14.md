**対象プロジェクト**: 家計 AI エージェント MVP（`intent-001-household-finance-ai-agent`）  
**目的**: 7/14に実践した **Code Generation L-2〜L-5**（実装・承認・PR マージ）を ChatGPT の解説で深く理解し、強い納得感を得る

---

## 使い方

1. 下の **推奨順** に従い、優先度の高いプロンプトから投げる
2. 各プロンプトの **「ChatGPT に投げる文面」** をコピー＆ペーストする
3. 初回のみ **補足コンテキスト** を先に渡すと、家計 AI プロジェクトの文脈で答えてもらいやすい
4. 返答を読んだら、自分の言葉で1〜2文要約してから次へ進む

### 優先度の意味

| 優先度 | 意味 |
|---|---|
| **最高** | 最初に理解すべき。7/14の「何をやった日か」と Layer ループのなぜがつかめる |
| **高** | 本日の実践の核心。承認・トレーサビリティ・各 Layer の意味が腹落ちする |
| **中** | 深掘り・次アクション前の理解。上記を押さえたあとに投げると効果的 |

## プロンプト一覧

### 1. Code Generation では具体的に何をするのか（7/14の位置づけ）

- **優先度**: 最高
- **WORK-LOG 対応**: §概要、§5 Layer 一覧、§10 全体像
- **なぜこのプロンプトを投げるか**: 7/14は設計日ではなく **実装日**だった。Discovery / Inception / Construction 設計との違いを最初に俯瞰しないと、Layer 承認が「ただのチェック」に見えてしまう。
- **理解が重要な理由**: **Code Generation 全体像**を押さえると、「なぜ今日 L-2〜L-5 を一気に回せたか（7/5で Plan まで承認済みだったから）」が納得できる。7/14のポイントは「小さく書いて止める」ことだと分かる。

**ChatGPT に投げる文面**:

```text
AI-DLC Construction の Code Generation では、具体的にどのような処理・判断を行うのか説明してください。

次の違いを明確にしてください:
- Inception（何を作るか・どう見せるかを文書化する）
- Construction 設計（Functional / NFR / Infrastructure / Code Generation Plan）
- Code Generation（Layer 単位でコードとテストを書く）

家計 AI（U-1）で L-2〜L-5（localStorage、Zaim、チャット、設定）を1日で実装した例を使い、
「なぜ設計承認後でないと一気に Layer を回せないか」も述べてください。
```

### 2. Layer ループでは具体的にどのような処理を行うのか

- **優先度**: 最高
- **WORK-LOG 対応**: §10 Layer ループ、`aidlc-code-generation-lite` / `aidlc-build-and-test-lite`
- **なぜこのプロンプトを投げるか**: 本日の作業の主動作が Layer ループだった。2 Skill がセットになる理由を理解するため。
- **理解が重要な理由**: **Layer ループ**を理解すると、「実装だけ」「テストだけ」では不十分で、diff-summary / build-and-test-report / review の3点セットが説明責任になる理由が分かる。7/14のポイント（小さく書いて承認する）の中核。

**ChatGPT に投げる文面**:

```text
AI-DLC の Code Generation における Layer ループでは、具体的にどのような処理を行うのか説明してください。

次の流れを、各ステップの目的つきで解説してください:
前 Layer 承認 → 実装（約5〜8ファイル + テスト）→ npm test / build →
diff-summary・build-and-test-report・review 作成 → 人間チェックポイントで停止 → 「L-n を承認」

なぜファイル数を制限するのか、なぜ Build/Test Report を Layer ごとに残すのかも教えてください。
家計 AI の L-3（Zaim）→ L-4（チャット）の連携例があると助かります。
```

### 3. 「L-n を承認」はなぜ状態機械の遷移コマンドなのか

- **優先度**: 最高
- **WORK-LOG 対応**: §9 承認の副作用、§10「承認」が持つ3つの意味、§11 パターン
- **なぜこのプロンプトを投げるか**: 本日いちばん短いプロンプトが最も強い効果を持った。「OK」ではなく遷移であることを腹落ちさせるため。
- **理解が重要な理由**: **承認ゲート**を理解すると、人間の役割が「実装者」より「遷移の許可者」であることが分かる。Inception の「requirements を承認」と同型で、Construction では直後にコードが増える点が違いとして理解できる。

**ChatGPT に投げる文面**:

```text
AI-DLC Construction でユーザーが「L-3 を承認」と短く送ると、Agent 側では具体的にどのような処理が起きるのか説明してください。

次の3つの意味に分けてください:
1. 過去の確定（review / Decision の更新）
2. 次の許可（次 Layer 実装の着手権）
3. 記録の更新（state / CODE_SUMMARY / traceability）

「単なる感想の OK」と「状態機械の遷移コマンド」の違いも明確にしてください。
家計 AI で L-3 承認後に L-4（/api/chat）実装が始まった例を使ってください。
```

### 4. Code Generation Plan が実装のレールになるとはどういうことか

- **優先度**: 高
- **WORK-LOG 対応**: §1〜§4 各 Layer、§9「7/5で固めた Plan」
- **なぜこのプロンプトを投げるか**: 7/14がスムーズだった主因は 7/5の Plan 承認（D-30）。Plan の価値を「後から」理解するため。
- **理解が重要な理由**: **Code Generation Plan**を理解すると、「なぜ L-6 を今日やらなかったか」「なぜ各 Layer のファイル一覧が最初から決まっていたか」が説明できる。実装日の迷いを減らせる。

**ChatGPT に投げる文面**:

```text
AI-DLC の Code Generation Plan（Layer ID、対象 Story/FR、予定ファイル、テスト方針、Approval）は、
実装日にどのような役割を果たすのか説明してください。

「Plan がないまま Agent に全部実装させる」場合と比較し、メリット・リスクを整理してください。
また家計 AI の例で:
- L-3 に zaim-parser と ImportForm が入っている理由
- L-6 Moneytree が Plan 対象外（後回し）である理由
が、実装判断にどう効いたかも述べてください。
```

### 5. L-2（localStorage）では何を固め、なぜ先に置くのか

- **優先度**: 高
- **WORK-LOG 対応**: §1 L-2、FR-3 / NFR-5、D-21
- **なぜこのプロンプトを投げるか**: 取り込み・チャット・削除の土台が L-2。永続化境界を最初に実装する意図を理解するため。
- **理解が重要な理由**: **L-2 localStorage**を理解すると、「サーバーに家計データを送らない」方針がコード構造（クライアント永続 + API は要約のみ）にどう落ちるかが分かる。プライバシー方針とレイヤ順の接続が見える。

**ChatGPT に投げる文面**:

```text
AI-DLC Code Generation の L-2 で localStorage アダプタとドメインモデル（HouseholdDataset / Transaction）を先に実装する理由を説明してください。

次も含めてください:
- FR-3 / NFR-5（端末内のみ永続）との対応
- Quota Guard が必要な理由
- DatasetSummary とホーム Empty/Success の関係
- L-3 取り込み・L-4 チャット・L-5 削除が、なぜ L-2 に依存するか

「永続化を後回しにして UI から作る」場合の失敗例も対比してください。
```

### 6. L-3（Zaim 取り込み・カテゴリ正規化）のポイントは何か

- **優先度**: 高
- **WORK-LOG 対応**: §2 L-3、BR-5 / BR-7 / BR-11、H-1 / SM-2
- **なぜこのプロンプトを投げるか**: 今日初めて「仮説（カテゴリ補正）→ コード（normalizer）」が接続された。MVP の仮説検証入口を理解するため。
- **理解が重要な理由**: **L-3 Zaim 取り込み**のなかで特に重要な **カテゴリ正規化（BR-7）** を理解すると、H-1 / SM-2（±10%）が「将来の AI 回答精度」だけでなく「入力データ品質」から始まることが分かる。L-3 のポイントが腹落ちする。

**ChatGPT に投げる文面**:

```text
家計 AI エージェントの L-3（Zaim CSV 取り込み）では、具体的にどのような処理を行うのか説明してください。

特に次を深掘りしてください:
1. 必須列検証（日付・金額・内容・カテゴリ）と Validation Error
2. カテゴリ正規化（食費 / 日用品 / 固定費 / その他）が H-1・SM-2・NFR-3 とどうつながるか
3. 再取り込みで全置換する理由（単一データセット）
4. Moneytree を UI で disabled にする（D-23 / BR-11）意味

「パースだけして正規化を後回し」にすると、後続の AI 相談で何が壊れるかも述べてください。
```

### 7. L-4（Bプランと `/api/chat`）では何を守っているのか

- **優先度**: 高
- **WORK-LOG 対応**: §3 L-4、D-17、transaction-summarizer / structured-response
- **なぜこのプロンプトを投げるか**: L-4 は外部 API 境界。Inception の Bプラン決定がコード構造になった瞬間を理解するため。
- **理解が重要な理由**: **L-4 `/api/chat`**のなかで特に重要な **「質問 + 関連取引要約のみ送信」** を理解すると、プライバシー（D-17）・Server-only Secret・構造化応答（改善案+良かった点）が一気につながる。L-4 のポイントが納得できる。

**ChatGPT に投げる文面**:

```text
AI-DLC で実装した家計 AI の L-4（チャット）について、Bプラン（質問文 + 関連取引要約のみを OpenAI に送る）が
コード構造にどう落ちるのか説明してください。

次のコンポーネントの責任分担も述べてください:
- transaction-summarizer（クライアント側で要約）
- /api/chat（キー秘匿・タイムアウト・mock）
- structured-response（improvements / positives 必須）
- ChatView（データ未取り込み時は誘導）

「全取引 raw を API に送る実装」と比べて、何を守り、何をトレードオフにしているかも整理してください。
```

### 8. L-5（全削除）で確認ダイアログが必須な理由

- **優先度**: 高
- **WORK-LOG 対応**: §4 L-5、BR-10、S-8 / FR-10 / SCR-4
- **なぜこのプロンプトを投げるか**: 破壊的操作は機能としては小さいが、信頼・プライバシーの要。ゲートの意味を理解するため。
- **理解が重要な理由**: **L-5 全削除**のなかで特に重要な **確認ダイアログ必須（BR-10）** を理解すると、「端末内保存」アプリでも誤削除リスクがあること、Story S-8 がなぜ独立しているかが分かる。L-5 のポイントが理解できる。

**ChatGPT に投げる文面**:

```text
家計 AI の L-5（設定・全データ削除）では、具体的にどのような処理・ガードを行うのか説明してください。

特に次を含めてください:
- DeleteConfirmDialog（alertdialog）が必須である理由（BR-10）
- キャンセル時にデータが残ることの受け入れ条件
- deleteAllHouseholdData が L-2 の clear をどう使うか
- 削除成功後にホーム Empty に戻るまでのユーザー体験

「削除ボタン一発」実装が AI-DLC / プロダクト仮説（端末内プライバシー）の観点からなぜ不適切かも述べてください。
```

### 9. トレーサビリティが「文書→コード」に着地するとはどういうことか

- **優先度**: 中
- **WORK-LOG 対応**: §10 トレーサビリティ、§11 Layer ごとのコード成果物マップ
- **なぜこのプロンプトを投げるか**: Inception でつないだ一本線が、今日初めてソースパスに落ちた。AI-DLC の核心が「実装日」に効く瞬間を味わうため。
- **理解が重要な理由**: **トレーサビリティのコード着地**を理解すると、PR レビューや学習時に「このファイルはどの要件か」を説明できる。次の Intent でも Layer 承認ごとに traceability を更新する動機が持てる。

**ChatGPT に投げる文面**:

```text
AI-DLC におけるトレーサビリティ（Hypothesis → Success Metric → Requirement → Story → Screen → Unit → Layer → ソースファイル）が、
Code Generation 完了時にどう「着地」するのか説明してください。

次の家計 AI の例を、上流から下流へ具体的に辿ってください:
1. H-1 → SM-2 → NFR-3 → BR-7 → L-3 category-normalizer.ts
2. FR-1 / S-1 / SCR-2 → L-3 ImportForm.tsx
3. FR-4〜9 / SCR-3 → L-4 ChatView.tsx + /api/chat
4. FR-10 / S-8 / SCR-4 → L-5 SettingsPanel.tsx

「実装後にトレースを付け直す」のと「Layer 承認ごとに更新する」の違いも述べてください。
```

### 10. 「次は実行しない」＋ Git で固める判断はなぜ重要か

- **優先度**: 中
- **WORK-LOG 対応**: §6 Git、§7 最終状態、§10「次へ進まない」、P-7 / P-8
- **なぜこのプロンプトを投げるか**: 本日の上手い運用だった。Harness 既定の自動前進と、人の意図（PR で定着）を両立する書き方を学ぶため。
- **理解が重要な理由**: **承認に制約を付けるプロンプト**を理解すると、Agent を止められずに全部走らせる失敗を防げる。実装完了・ゲート通過・次レビュー開始・main 定着を分けて考える習慣が身につく。

**ChatGPT に投げる文面**:

```text
AI-DLC Construction で、ユーザーが
「L-5 を承認するがまだ次は実行しないで。GitHub にプッシュして PR 作成」
と指示した場合、Agent / Harness の観点で何が起き、何が起きてはいけないのか説明してください。

次を対比してください:
A. 「L-5 を承認」（だけ）→ UI Review や Unit Build-and-Test に自動で進む可能性
B. 「承認 + 次は実行しない + PR」→ Decision/state 更新と Git 定着のみ

なぜ「実装完了」と「UI Review 開始」と「main マージ」を同日でも分離すべきなのか、
リスク管理と学習（振り返りログ作成）の両面から述べてください。
```

6. プロンプト **9 → 10**
