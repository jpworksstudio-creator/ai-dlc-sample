## 11. AI-DLC アプリケーション開発の理解（7/5実践で深まった点）

### Construction の本質（実装より前に設計を固める）

- Construction は「すぐコードを書く工程」ではなく、**実装可能性を段階的に閉じる工程**だった
- 実際の順序は `Input Readiness → Plan → Functional → NFR → NFR Design → Infrastructure → Code Generation` で、各段階に承認ゲートがある
- そのため、L-1 実装時点で「何を作るか／何を作らないか（L-6後回し）」が明確だった

### 承認ゲートの価値

- 「未承認のまま次工程へ進まない」ルールが、スコープ拡大と手戻りを抑えた
- 特に技術選定変更（Next.js 15 想定 → 16.2.10）を、設計フェーズで吸収できたのは大きい
- Decision（D-19〜D-30）が積み上がることで、後続 Layer の判断根拠が文書として残った

### トレーサビリティの実効性

- 7/5時点で「要件→設計→Layer」の接続が始まり、`traceability.md` に L-1 実装根拠を反映できた
- これにより、レビュー時に「この実装がどの要件由来か」を確認しやすくなった

### 学習メモ（次回に活かす）

- Code Generation Plan は「ファイル数（5〜8）」「テスト方針」「承認単位」を先に固定すると進行が安定する
- Build/Test Report と Layer Review を同時に作る運用が、実装品質の説明責任を強くする
